# Workflow Handbook

Actionable patterns for agent prompting, execution scaffolds, context management, and recovery.
Last updated: 2026-03-19 (from research2.md — OpenHands, SWE-Agent, Aider, Devon, Devin, Claude/Cursor)

---

## Worker Execution Scaffold

### Canonical Loop: EXPLORE → PLAN → EDIT → VERIFY → REPORT
All major production systems (OpenHands, SWE-Agent, Aider, Claude/Cursor plan mode) converge on this or equivalent.

1. **EXPLORE** — read only the files and tests relevant to the task; build mental model
2. **PLAN** — produce a compact plan (≤6 short steps) before the first edit
3. **EDIT** — make minimal changes only in allowed files; prefer existing project patterns
4. **VERIFY** — run listed validation commands; check acceptance criteria explicitly
5. **REPORT** — structured RESULT mapped to acceptance criteria + remaining risks

### Failure Recovery Protocol
After 2 failed verification cycles, emit a structured `FAILURE_REPORT` (do not thrash):
```
FAILURE_CLASS: <category>
TOP_3_HYPOTHESES: <ranked causes>
BEST_NEXT_ACTION: <concrete next step>
NEEDED_CONTEXT_OR_TOOL: <what would unblock>
SHOULD_ESCALATE: yes|no
```
Never repeat a failed command unchanged. Changing approach is required before re-try.

### Self-Monitoring Rules (workers)
- Stay in scope: only modify `allowed_files`; emit `ESCALATE` if scope extension needed
- Minimize diff: prefer existing abstractions over new ones
- Verify explicitly: map each check to an acceptance criterion, don't assume "looks right"
- Delete temp files: unless the task explicitly says to keep them
- Stop when stuck: fail with a structured report, not a bad speculative fix
- Do not push, merge, open PRs, or mutate coordinator state except through `mac10`/`codex10` CLI

## Prompt Stack Architecture

### Five-Layer Stack (keep layers separate)
1. **Role invariant** — small, static, checked into git; one file per role; no live task data
2. **Repo rules / memory summary** — short, mostly static; knowledge files, pitfalls, patterns
3. **Dynamic task packet** — generated by coordinator per session/worktree (see task packet format below)
4. **Execution scaffold** — small reusable EXPLORE/PLAN/VERIFY fragment; injected with every task
5. **Recovery epilogue** — injected only after failure or compaction

**Critical rule**: stable instructions and live task state must never be in the same file.
The current worker loop instructions are correct; do not embed active task details into shared knowledge files.

### Task Packet Format (coordinator-generated per session)
```yaml
task_id: T-<id>
request_id: req-<id>
base_commit: <sha>
worktree: .worktrees/wt-N
role: worker
goal: "<one-sentence goal>"
allowed_files:
  - <path1>
  - <path2>
forbidden_paths:
  - <glob>
validation:
  - <shell command>
acceptance_criteria:
  - <verifiable condition>
budget:
  max_turns: 12
  max_failed_verifications: 2
escalate_when:
  - need_file_outside_scope
  - missing_dependency_or_broken_env
  - repeated_failure
deliverables:
  - changed_files
  - validation_results
  - remaining_risks
```

## Role Boundary Guidelines

### Interface Agent (Master-1)
- Strictly code-blind; never reads or writes code
- Only translates user intent → `mac10 request/fix` and surfaces results back

### Architect (Master-2)
- Mostly read-only and semantic: classify, decompose, define task packets
- Move backlog thresholds, timers, reset counters out of the prompt and into Node/SQLite
- Return parseable YAML decomposition (TIER, TASKS with id/title/goal/domain/allowed_files/validation/dependencies)

### Allocator (Master-3)
- Should be as thin/deterministic as possible; allocation is a state-machine problem
- Hard rules first: capability match, concurrency limit, domain affinity, worktree ownership, retry counts
- LLM routing only as last resort after hard rules

### Workers
- Follow EXPLORE → PLAN → EDIT → VERIFY → REPORT scaffold exactly
- Domain-locked per session: one domain, one PR, then exit
- Parallelize by file/domain boundary only — parallel same-file = conflict trap

## Context Management

### Always-On vs On-Demand
- Keep role invariant + repo rules short and static (always loaded)
- Load domain knowledge, research rollups, and file content on-demand only
- Never include live task data in always-loaded files (cross-worker contamination risk)
- More always-on context = lower adherence (confirmed across Claude, Cursor, Aider, OpenHands)

### Knowledge File Token Budgets
- `codebase-insights.md` ≤ 2000 tokens
- `patterns.md` ≤ 1000 tokens
- Per-domain README ≤ 1000 tokens
- Handbook files: no strict limit but keep concise; remove redundancy aggressively

### Summarization / Compaction
- When compacting: preserve acceptance criteria, touched files, and known-bad hypotheses — these are the facts most likely to cause regression if lost
- Prefer structured summaries over raw transcript dumps: goal, files changed, commands/tests run, key pattern learned, failed approach to avoid, unresolved risk

---

## Parallelism Guidelines

### When to Parallelize
- Different files, different domains, independent acceptance criteria → safe to parallelize
- Same-component tasks within one request → serialize with `depends_on`
- Any task touching global-risk files (lockfiles, schema.sql, db.js, cli-server.js) → serialize

### Parallelism is Not Free
- Each additional worker context window costs tokens + coordination overhead
- Parallel agents with overlapping file scope = guaranteed merge conflict risk
- Prefer focused sequential sessions over many parallel workers for tightly coupled work

## Recovery Patterns

### Prompt-Level (inject after failure)
OpenHands-style recovery prompt:
> After repeated failure, step back. List 5–7 possible root causes. Rank by likelihood. Address the most likely. If the current plan is invalidated, propose a new plan rather than hacking around the obstacle.

### Harness-Level (coordinator-owned)
- Watchdog: tiered escalation (warn → nudge → triage → terminate) based on heartbeat + progress age
- Circuit breaker: if iteration count grows while diff stays tiny (same files, same edits) → fail fast
- Hook points: `pre_task_start`, `post_action`, `post_failure`, `pre_compaction`, `heartbeat_late` — inject short observations back into agent loop
- Coordinator enforces: no-overlap scheduling, retry ceilings, heartbeat deadlines, merge gates, allowed tool scopes
