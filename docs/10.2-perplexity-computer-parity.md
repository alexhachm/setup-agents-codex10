# mac10 10.2 — Perplexity & Computer Parity Gap Analysis

> Tracking document for feature gaps identified via research rollups and gap analysis.
> Last updated: 2026-04-16

## Overview

This document tracks the gap between mac10's current capabilities and parity with comparable multi-agent coding systems. Gaps are sourced from research rollups in `.codex/knowledge/research/topics/`.

## Architecture Gaps

### GAP-A1: Monolithic db.js (SRP violation)
- **Source**: coordinator-core rollup, clean-run rollup (SOLID analysis)
- **Description**: `db.js` (~4,700 LOC) handles persistence, lifecycle, telemetry, research, and project-memory. Violates Single Responsibility Principle.
- **Impact**: Hard to test, maintain, and extend independently
- **Recommendation**: Extract into service layer (requestService, taskService, workerService, etc.) + repository layer + migration layer
- **Status**: PLANNED
- **Notes**: Architectural refactor tracked for 10.3. Current state is stable and functional.

### GAP-A2: Monolithic cli-server.js (transport + domain coupling)
- **Source**: coordinator-core rollup, coordinator-routing rollup
- **Description**: `cli-server.js` (~4,800 LOC) mixes transport (Unix socket), command dispatch, domain logic, and routing decisions.
- **Impact**: Routing policy changes require touching transport code
- **Recommendation**: Extract RoutingService, CommandSchema validation, and domain handlers into separate modules
- **Status**: PLANNED

### GAP-A3: Dual Protocol Drift (DB + handoff.json)
- **Source**: coordinator rollup
- **Description**: Schema defines DB + mail table as source of truth, but `cli-server.js` still writes `handoff.json` and touches signal files for legacy compatibility.
- **Impact**: Ghost bugs from state divergence between DB and file-based protocols
- **Recommendation**: Quarantine legacy handoff bridge with explicit toggle: "DB/mail only" or "compatibility mode"
- **Status**: PARTIAL — handoff bridge identified but not yet quarantined

### GAP-A4: No Centralized State Machine
- **Source**: coordinator-core rollup, status rollup
- **Description**: Status field values and allowed transitions are scattered across schema and code. No formal state machine.
- **Impact**: Invalid state transitions possible; recovery logic duplicated
- **Recommendation**: Create StatusMachine class with transition maps for each entity type
- **Status**: PLANNED

## Operational Gaps

### GAP-O1: setInterval in Async Loops
- **Source**: final-live rollup (setTimeout vs setInterval research)
- **Description**: Allocator, watchdog, and merger use `setInterval` for their loops. With async operations, `setInterval` doesn't wait for task completion, risking overlap and queue buildup.
- **Impact**: Potential for concurrent loop ticks processing the same state
- **Recommendation**: Refactor to recursive `setTimeout` pattern
- **Status**: PLANNED
- **Files**: `allocator.js:16`, `watchdog.js` (interval setup)

### GAP-O2: No Deterministic Fallback Scheduler
- **Source**: coordinator rollup, coordinator-routing rollup
- **Description**: Task assignment depends on the allocator agent being alive. No built-in fallback if the allocator role crashes.
- **Impact**: Ready tasks pile up when allocator is down
- **Recommendation**: Implement deterministic fallback assigner (priority + dependency + affinity) that activates after allocator inactivity
- **Status**: PLANNED

### GAP-O3: No Connect Timeout for Inter-Process Communication
- **Source**: node-http rollup
- **Description**: Node.js has no native "connect timeout." Coordinator → worker communication could hang on dead connections.
- **Impact**: Stalled operations when workers crash mid-connection
- **Recommendation**: Implement `httpRequestWithTimeouts` wrapper with explicit connect (2-3s), socket (5-15s), and total (10-30s) timeouts via AbortController
- **Status**: PLANNED

### GAP-O4: Daemon Mode
- **Source**: infra rollup
- **Description**: No systemd/launchd service files for running coordinator as a persistent daemon.
- **Impact**: Coordinator must be manually started; no auto-restart on crash
- **Recommendation**: Create service files for systemd (Linux) and launchd (macOS)
- **Status**: FIXED (10.2)
- **Files**: `services/mac10-coordinator.service`, `services/com.mac10.coordinator.plist`

## Extension Gaps

### GAP-E1: No Formal Plugin Registry
- **Source**: coordinator-extensions rollup
- **Description**: Extensions are ad-hoc hooks in `index.js`, not formal extension points. No registry, no lifecycle management.
- **Impact**: Hard to add new backends or agent types without editing core
- **Recommendation**: Create ExtensionRegistry class with well-defined hooks (onTaskAssigned, onTaskCompleted, onWorkerSpawned, etc.)
- **Status**: PLANNED

### GAP-E2: Knowledge Staleness Tracking
- **Source**: coordinator-extensions rollup
- **Description**: No explicit refresh policy for knowledge overlays. Overlays may inject stale research context.
- **Impact**: Workers receive outdated domain knowledge
- **Recommendation**: Add TTL/refresh policies to knowledge-metadata.js
- **Status**: PARTIAL — staleness detection added (>10 changes warning), but no auto-refresh

### GAP-E3: Model Routing Coupled to CLI Layer
- **Source**: coordinator-extensions rollup, coordinator-routing rollup
- **Description**: Model selection logic (fast/deep/economy → model IDs) lives in `cli-server.js` instead of a separate service.
- **Impact**: Can't change model routing without touching transport code
- **Recommendation**: Extract model-routing logic into ModelRoutingService
- **Status**: PLANNED

## Infrastructure Gaps

### GAP-I1: Backend Convergence
- **Source**: infra rollup
- **Description**: tmux, Docker, and msb backends have different liveness semantics. No unified abstraction.
- **Impact**: Backend-specific bugs; hard to add new backends
- **Recommendation**: Create WorkerBackend interface with unified `createWorker()`, `killWorker()`, `isAlive()`, `getOutput()`
- **Status**: PARTIAL — `worker-backend.js` exists but doesn't fully abstract liveness

### GAP-I2: Worktree Lifecycle Management
- **Source**: infra rollup
- **Description**: Worktree creation, branch tracking, and cleanup are not centralized.
- **Impact**: Orphaned worktrees accumulate; branch drift during integration
- **Recommendation**: Centralize in WorktreeManager with create/track/cleanup lifecycle
- **Status**: PLANNED

### GAP-I3: Generated-File Cleanup Policy
- **Source**: git-generated-noise-policy rollup, workspace-hygiene-wrapper rollup
- **Description**: Runtime generates state files, logs, and research artifacts inside the repo. No cleanup policy.
- **Impact**: Git status noise, accidental commits of runtime artifacts
- **Recommendation**: Three-tier layout (source/generated/ephemeral), cleanup hooks on startup
- **Status**: PARTIAL — .gitignore updated in 10.2 to cover runtime artifacts

### GAP-I4: Settings File Security
- **Source**: Security best practices
- **Description**: Settings files may contain API keys. No .gitignore protection.
- **Impact**: Risk of committing secrets
- **Recommendation**: Add .gitignore entries for `.env`, `*.secret`, `credentials.json`, `settings.local.json`
- **Status**: FIXED (10.2)
- **Files**: `.gitignore`

## Documentation Gaps

### GAP-D1: README Advertises Disabled Features
- **Source**: coordinator rollup
- **Description**: README references web dashboard at http://localhost:3100, but GUI is disabled in `index.js` (lines 11-12).
- **Impact**: User confusion; wasted debugging time
- **Recommendation**: Update README to mark dashboard as legacy/planned, or remove references
- **Status**: FIXED (10.2)

### GAP-D2: No Architecture Document
- **Source**: coordinator-core rollup
- **Description**: No standalone architecture document. System understanding requires reading code.
- **Impact**: High onboarding cost for new operators
- **Recommendation**: Create `docs/current-architecture.md` with component breakdown, state machines, and design decisions
- **Status**: FIXED (10.2)
- **Files**: `docs/current-architecture.md`

### GAP-D3: No Built-In Skills for Common Operations
- **Source**: coordinator-extensions rollup
- **Description**: No pre-packaged Claude Code skills for status checking, merge troubleshooting, research management.
- **Impact**: Operators must know CLI commands from memory
- **Recommendation**: Create built-in skill files for coordinator-status, merge-troubleshooting, research-queue, worker-lifecycle, knowledge-management
- **Status**: FIXED (10.2)
- **Files**: `templates/skills/coordinator-status/SKILL.md`, etc.
