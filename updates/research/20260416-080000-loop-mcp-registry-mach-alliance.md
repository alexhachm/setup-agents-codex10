# Topic
MCP protocol ecosystem updates: MACH Alliance MCP Registry

## Sources (URLs)
- https://machalliance.org/mach-alliance-mcp-registry

## Relevance to 10.2
If MCP becomes the de facto “connector/skills transport,” registry patterns (discovery, metadata standardization, governance signals) become critical to feature parity: users will expect “browse + add tools” workflows, plus enterprise allowlisting/verification.

## Findings
- MACH Alliance describes an **MCP Registry** as a central directory of MCP-compliant servers for safe discovery/integration by MCP clients (agents, IDE plug-ins, orchestration tools). [MACH Alliance MCP Registry](https://machalliance.org/mach-alliance-mcp-registry)
- The page emphasizes **vendor-neutral** operation and highlights registry functions: register/describe servers, discover capabilities, apply governance & trust policies, and standardize metadata. [MACH Alliance MCP Registry](https://machalliance.org/mach-alliance-mcp-registry)
- It explicitly calls out enterprise use: curated catalog aligned with internal policy, ownership/support metadata, optional verification/review signals, and separation of public vs internal entries. [MACH Alliance MCP Registry](https://machalliance.org/mach-alliance-mcp-registry)
- The registry is described as exposing an **API** and metadata format aligned with MCP so clients can query/filter and bootstrap configuration. [MACH Alliance MCP Registry](https://machalliance.org/mach-alliance-mcp-registry)

## Recommended Action
- For mac10 10.2 parity: add a roadmap item for “connector discovery via registry” (search/filter by capability), and an enterprise control plane concept (allowlist + verification badges + ownership fields + public/internal partitioning).
